# 001 — Multi-Process Frame Rate Halving / 多进程帧率降半

> **Date**: 2026-04-01
> **Status**: Resolved
> **Tags**: `D3D11`, `DXGI`, `Windows`, `DWM`, `Sleep`, `Timer`

## Problem Description / 问题描述

**Symptom**: When the wallpaper process and a D3D11 window-mode process run simultaneously (on different monitors), the wallpaper frame rate drops from the target 60fps to ~32fps. The window-mode process maintains 60fps normally.

**Reproduction Steps**:
1. Launch wallpaper mode: `run-wallpaper-exe.bat` (`--wallpaper --debug --shader ... --monitor 0`)
2. Launch D3D11 window mode: `run-d3d11-exe.bat` (`--d3d11 --shader ...`)
3. Maximize the window-mode window to fullscreen on a different monitor
4. Observe the wallpaper debug overlay: FPS drops from ~60 to ~32

**Expected Behavior**: Wallpaper maintains target 60fps regardless of other processes.

**Actual Behavior**: Wallpaper FPS drops to ~32 (exactly half of 60). Key debug values:
- RenderTime: 0.172ms (GPU rendering is extremely fast)
- Frame Time: 30.682ms (exactly ~30fps interval)
- adaptive: 60.0, target: 60 (adaptive FPS did not lower)

The huge gap between RenderTime (0.172ms) and Frame Time (30.682ms) indicates the bottleneck is in the **frame pacing/sleep logic**, not GPU rendering.

## Environment / 环境信息

- **OS**: Windows 11 (24H2+)
- **GPU**: Any (issue is OS-level, not GPU-specific)
- **Build Config**: Release, MSVC, Ninja
- **Rendering**: Both processes use D3D11 with `DXGI_SWAP_EFFECT_FLIP_DISCARD`
- **Key Detail**: Two independent processes, each with its own DXGI Flip Model SwapChain

## Investigation / 排查过程

### Attempt 1: Filter own-process windows in occlusion detection
- **What was tried**: Added `GetCurrentProcessId()` filtering in `IsMonitorOccluded` and `IsFullscreenAppRunning` to skip the current process's own windows, preventing the window-mode SDL window from triggering wallpaper occlusion/pause.
- **Result**: ❌ Did not fix
- **Observation**: The two instances are **separate processes** with different PIDs. The occlusion detection already wouldn't match them. The root cause is elsewhere.

### Attempt 2: Fix `IsFullscreenAppRunning` multi-monitor detection
- **What was tried**: Changed `IsFullscreenAppRunning` from using `SM_CXSCREEN/SM_CYSCREEN` (primary monitor only) to iterating `wallpaperWindows` and checking if the foreground window covers any wallpaper monitor.
- **Result**: ❌ Did not fix (but was a valid bug fix for a separate issue)
- **Observation**: This fixed a real bug in multi-monitor fullscreen detection, but the frame rate halving persisted.

### Attempt 3: Enable `DXGI_PRESENT_ALLOW_TEARING`
- **What was tried**: Added tearing support to bypass DWM VSync throttling:
  - `IDXGIFactory5::CheckFeatureSupport(DXGI_FEATURE_PRESENT_ALLOW_TEARING)` in `Init()`
  - `DXGI_SWAP_CHAIN_FLAG_ALLOW_TEARING` in `AddSwapChain()`
  - `DXGI_PRESENT_ALLOW_TEARING` flag in `Present()` when `syncInterval=0`
  - Matching flags in `ResizeBuffers()`
- **Result**: ❌ Did not fix
- **Observation**: `DXGI_PRESENT_ALLOW_TEARING` requires the window to be in the **foreground**. Wallpaper windows are embedded in WorkerW (always behind all windows), so this flag silently fails or falls back to normal behavior. However, this change was kept as it's still beneficial for window-mode D3D11.

### Attempt 4: `SetMaximumFrameLatency(1)` + High-resolution waitable timer
- **What was tried**:
  1. Set `IDXGIDevice1::SetMaximumFrameLatency(1)` to reduce DXGI frame queue depth
  2. Replace `SDL_Delay()` with `CreateWaitableTimerExW(CREATE_WAITABLE_TIMER_HIGH_RESOLUTION)` + spin-wait
- **Result**: ✅ Fixed!
- **Observation**: The high-resolution waitable timer is not affected by the background process timer resolution degradation. Frame rate restored to 60fps.

## Root Cause / 根因分析

The issue has **two contributing factors**:

### Factor 1: Windows 10 2004+ `timeBeginPeriod` behavior change

Starting from Windows 10 version 2004, `timeBeginPeriod()` became **process-level** instead of global. This means:
- SDL2 calls `timeBeginPeriod(1)` during initialization (in `SDL_TicksInit()`)
- This sets the timer resolution to 1ms **for the calling process only**
- When the wallpaper process is a **background process** (not foreground), Windows power management may **ignore** its `timeBeginPeriod(1)` setting
- `Sleep(N)` / `SDL_Delay(N)` precision degrades from ~1ms to ~15.625ms (default tick period)

### Factor 2: `SDL_Delay(16)` oversleeping in background processes

The original frame rate control code:
```cpp
SDL_Delay(static_cast<Uint32>((targetFrameTime - frameElapsed) * 1000.0f));
// With targetFPS=60, frameElapsed≈0.2ms: SDL_Delay(16)
```

When timer precision is 15.625ms (default tick), `Sleep(16)` rounds up to the next tick boundary:
- Best case: 15.625ms (one tick) → ~62fps ✅
- Worst case: **31.25ms (two ticks)** → ~32fps ❌

The wallpaper process consistently hits the worst case when another D3D11 process is running, because:
1. The other process takes focus → wallpaper becomes background
2. Background process timer resolution degrades
3. `Sleep(16)` oversleeps to ~31ms
4. Frame rate drops to exactly half: 60fps → ~32fps

### Factor 3: DXGI Flip Model DWM throttling (secondary)

`DXGI_SWAP_EFFECT_FLIP_DISCARD` SwapChains go through DWM composition. When multiple Flip Model SwapChains exist across processes, DWM may throttle the non-focus process's frame queue. `SetMaximumFrameLatency(1)` helps reduce this effect.

## Solution / 解决方案

Three-pronged fix applied:

### Key Changes

| File | Change Description |
|------|--------------------|
| `src/d3d11_renderer.h` | Added `dxgi1_5.h` include, `tearingSupported_` member variable |
| `src/d3d11_renderer.cpp` | Tearing detection in `Init()`, `ALLOW_TEARING` flag in `AddSwapChain()`/`ResizeSwapChain()`/`Present()`, `SetMaximumFrameLatency(1)` |
| `src/main.cpp` | Replaced `SDL_Delay()` frame control with high-resolution waitable timer + spin-wait |

### Code Highlights

**1. High-resolution waitable timer (main.cpp)**
```cpp
// Phase 1: High-resolution timer wait (no CPU usage, ~0.5ms precision)
if (remaining > 0.002f) {
    static HANDLE hTimer = CreateWaitableTimerExW(
        nullptr, nullptr,
        CREATE_WAITABLE_TIMER_HIGH_RESOLUTION, TIMER_ALL_ACCESS);
    if (hTimer) {
        LARGE_INTEGER dueTime;
        dueTime.QuadPart = -static_cast<LONGLONG>((remaining - 0.001f) * 10000000.0f);
        SetWaitableTimerEx(hTimer, &dueTime, 0, nullptr, nullptr, nullptr, 0);
        WaitForSingleObject(hTimer, static_cast<DWORD>(remaining * 1000.0f) + 1);
    }
}
// Phase 2: Spin-wait for remaining ≤2ms
while (true) {
    Uint64 spinEnd = SDL_GetPerformanceCounter();
    float totalElapsed = static_cast<float>(spinEnd - now) / static_cast<float>(freq);
    if (totalElapsed >= targetFrameTime) break;
}
```

**2. DXGI Allow Tearing + SetMaximumFrameLatency (d3d11_renderer.cpp)**
```cpp
// Tearing detection
ComPtr<IDXGIFactory5> factory5;
if (SUCCEEDED(dxgiFactory_.As(&factory5))) {
    BOOL allowTearing = FALSE;
    factory5->CheckFeatureSupport(DXGI_FEATURE_PRESENT_ALLOW_TEARING,
                                   &allowTearing, sizeof(allowTearing));
    tearingSupported_ = (allowTearing == TRUE);
}

// Frame latency control
ComPtr<IDXGIDevice1> dxgiDevice;
if (SUCCEEDED(device_.As(&dxgiDevice))) {
    dxgiDevice->SetMaximumFrameLatency(1);
}
```

### Why the high-resolution waitable timer works

`CreateWaitableTimerExW` with `CREATE_WAITABLE_TIMER_HIGH_RESOLUTION` (Windows 10 1803+):
- Precision: ~0.5ms (uses platform timer, not `timeBeginPeriod`)
- **Not affected by background process timer resolution degradation**
- Does not consume CPU while waiting (thread is suspended)
- Fallback: If unavailable (pre-1803), falls back to `SDL_Delay` with reduced precision
