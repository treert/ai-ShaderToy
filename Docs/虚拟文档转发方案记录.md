# 虚拟文档 + 外部扩展转发方案记录

> 记录时间：2026-04-04  
> 状态：**已搁置**（外部 HLSL 扩展支持不足）

## 方案概述

**目标**：在 `.stoy` 文件的 HLSL 代码块（`code [=[ ... ]=]` 和 `common [=[ ... ]=]`）中，复用外部 HLSL 扩展的 Language Server 功能（hover、completion、go-to-definition 等）。

**核心机制**：
1. 通过 `vscode.workspace.registerTextDocumentContentProvider` 注册自定义 scheme（`stoy-hlsl://`）的虚拟文档
2. 每个 `.stoy` 文件的每个 HLSL 代码块生成一个虚拟 `.hlsl` 文档，内容 = 注入的 cbuffer/纹理声明 + common 代码 + pass code
3. 在 Language Client 的 middleware 中拦截 HLSL 区域的请求
4. 通过 `vscode.commands.executeCommand('vscode.executeHoverProvider', virtualUri, position)` 将请求转发给外部 HLSL 扩展
5. 将外部扩展返回的结果进行位置映射（虚拟行号 → 物理行号）后返回

## 虚拟文档结构

每个 pass 的虚拟 HLSL 文档按以下 8 层结构组装：

```
1. cbuffer ShaderToyUniforms : register(b0)    — 固定 208 字节
2. inner_vars 别名                              — static float3 iResolution = ...
3. cbuffer TextureParams : register(b1)         — TexelSize
4. 外部纹理声明                                  — Texture2D + SamplerState
5. Pass 输出纹理声明
6. common 代码（原样注入）
7. 用户 pass 代码（mainImage 函数）
8. main 入口包装                                — PS_INPUT → fragCoord → mainImage
```

## URI 设计

```
stoy-hlsl://hlsl/<blockType>/<blockName>/<base64url(physicalUri)>.hlsl
```

示例：
```
stoy-hlsl://hlsl/pass/Main/ZmlsZTovLy9jJTNBL1VzZXJzLy4uLg.hlsl
stoy-hlsl://hlsl/common/_common_/ZmlsZTovLy9jJTNBL1VzZXJzLy4uLg.hlsl
```

> 注意：最初使用 `encodeURIComponent` 编码物理 URI，导致双重编码问题（`%3A` → `%253A`），后改为 base64url 编码解决。

## 位置映射

```
physicalLine → virtualLine:  prefixLineCount + (physicalLine - blockRange.startLine)
virtualLine → physicalLine:  blockRange.startLine + (virtualLine - prefixLineCount)
```

- `prefixLineCount`：注入代码的总行数（cbuffer + 别名 + 纹理声明等）
- `blockRange`：代码块在物理文件中的行号范围

## 实现文件

已实现的相关代码（在 `stoy-vscode-plugin/src/` 中）：

| 文件 | 职责 |
|------|------|
| `virtualDocProvider.ts` | 虚拟文档 ContentProvider，管理缓存和 URI 解析 |
| `requestForwarder.ts` | 请求转发器，含 `ensureVirtualDocOpened` + 各 forward 方法 |
| `hlslGenerator.ts` | 生成虚拟 HLSL 文档内容（8 层结构） |
| `positionMapper.ts` | 物理行号 ↔ 虚拟行号双向映射 |
| `extension.ts` | middleware 拦截 + 位置映射回传 |

## 测试结果

### 测试环境
- VS Code
- 外部 HLSL 扩展：`slevesque.shader`、`TimGJones.hlsltools`

### `slevesque.shader`
- **结果：无效**
- 该扩展只提供 TextMate 语法高亮，没有 Language Server
- `executeHoverProvider` 返回 0 hovers

### `TimGJones.hlsltools`
- **结果：部分有效**
- 该扩展有真正的 HLSL Language Server，能响应虚拟文档的请求
- hover 和 completion 可以工作
- 但该扩展有报错，且已停止维护

## 搁置原因

1. **外部扩展依赖**：方案强依赖外部 HLSL 扩展的质量和对自定义 scheme 虚拟文档的支持
2. **可用扩展有限**：`slevesque.shader` 无 Language Server；`TimGJones.hlsltools` 有 LS 但已停更且有 bug
3. **Diagnostics 不可用**：VS Code API 限制，`executeCommand` 无法拉取虚拟文档的诊断信息
4. **用户安装门槛**：需要用户额外安装特定版本的 HLSL 扩展，增加使用复杂度

## 恢复条件

如果未来出现以下情况，可以考虑恢复此方案：

- 出现高质量、积极维护的 HLSL Language Server 扩展
- 该扩展对自定义 scheme 的虚拟文档有良好支持
- 或 VS Code API 放开了虚拟文档的 diagnostics 支持

恢复时参考上述实现文件和 URI 设计，核心代码已在 git 历史中保留（commit `991c833`）。

## 启用方式

该方案已通过配置项集成到插件中，不再需要手动修改代码。

### 配置切换

在 VSCode 设置中搜索 `stoy.hlsl.provider`，选择：

| 值 | 说明 |
|---|---|
| `builtin`（默认） | 内置正则方案：插件自带的 HLSL 函数签名库 + 正则符号扫描 |
| `externalLsp` | 外部 LSP 方案：通过虚拟文档转发给外部 HLSL 扩展 |

切换后需要**重载窗口**才能生效（插件会自动弹窗提示）。

### 使用 externalLsp 模式的前提

1. 安装一个支持 Language Server 的 HLSL 扩展（如 shader-validator）
2. 该扩展需要能响应 `stoy-hlsl://` scheme 的虚拟文档请求

### 实现架构

```
用户编辑 .stoy 文件
  → extension.ts middleware 拦截 HLSL 区域请求
  → VirtualDocProvider 提供虚拟 .hlsl 文档
  → RequestForwarder 通过 executeCommand 转发给外部扩展
  → 外部扩展返回结果 → 位置映射回物理文件 → 返回给用户
```

server 端在 `externalLsp` 模式下自动禁用 HLSL providers，避免两套方案结果混合。DSL providers 始终保持活跃。

## 替代方案

内置正则方案（`builtin` 模式）：
- 内置 HLSL 函数签名表（hover + completion）
- 解析用户代码中的符号（函数、变量、结构体定义）
- 不依赖任何外部扩展
